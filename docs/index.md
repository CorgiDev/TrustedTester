<h1 id="about-this-document">About This Document</h1>
<h2 id="who-should-use-this-document">Who Should Use This Document</h2>
<p>This document has been designed for and is intended for use by Trusted Testers.</p>
<p>A Trusted Tester is a person certified to provide accurate and repeatable Revised 508 conformance test results for web content. A Trusted Tester follows the Revised Section 508 Conformance Test Process for Web, uses approved testing tools, and evaluates web applications for conformance with Revised 508 standards. Trusted Testers are those who have passed the Trusted Tester Certification Exam.</p>
<p>For more information on the Revised 508 Trusted Tester Training Course and Exam, contact the Department of Homeland Security Office of Accessible Systems &amp; Technology (OAST) Accessibility Helpdesk at <a href="mailto:accessibility@dhs.gov">accessibility@dhs.gov</a>.</p>
<h2 id="harmonized-baseline-alignment">Harmonized Baseline Alignment</h2>
<p>This test process incorporates all tests in the &quot;Harmonized Processes for Revised 508 Testing: Baseline Tests for Web Accessibility&quot; version 3.0. The baseline tests established the minimum steps required to determine compliance with Revised 508 standards and WCAG 2.0 Level A and AA. Test instructions that are specific to Trusted Tester only are identified with *TT-specific* or &quot;[no baseline].&quot; The outcomes of these tests will be reflected only in Revised 508 test results.</p>
<p>Baseline test results will be reported separately and are not affected by Trusted Tester-specific tests.</p>

<h2 id="web-content-tests-only">Web Content Tests Only</h2>
<p>This test process covers web content only. Trusted Tester versions 4.x and older were developed for the original Section 508 standards, which had separate requirements for web and software. With the Revised 508 standards applying WCAG 2.0 Level A and AA to web, software and other electronic content, combining software and web in one test process was the original plan.</p>
<p>However, because the Revised 508 Standards has other requirements for software in addition to WCAG 2.0, and software testing tools were not yet available, the software test process will be separate. While it is not as common due to HTML5 capabilities, software elements are still found with web content in web applications. To test the software elements, use the software test process.</p>
<p>Similarly, any other operating systems, browser or platforms such as mobile tablets, must be evaluated using other testing procedures.</p>
<h2 id="testing-order">Testing Order</h2>
<p>The numbering of the tests within the test process do not necessarily indicate the order that tests must be performed. Each tester and each application may determine the optimal testing order for coverage and productivity.</p>
<p>It is recommended, however, that the first test performed is Test 1 Conforming Alternate Version. Identifying conforming alternate versions of content helps define the scope of testing and avoids unnecessary testing. Non-conforming content that has a conforming alternate version is excluded from testing.</p>
<p>Test 2 Auto-Playing and Auto-Updating Content and Test 3 Flashing are next in the test process, followed by Test 4 Keyboard Access and Focus. These test WCAG success criteria that are covered in <a href="https://www.w3.org/TR/UNDERSTANDING-WCAG20/conformance.html">Conformance Requirement 5 Non-Interference</a>. Failure to meet these success criteria could interfere with any use of the page and may indicate critical accessibility issues.</p>
<p>The test process was designed to streamline the sequence for testers, however testers may choose to do the tests (after Test 1) in their preferred order. Each Test Condition (after Test 1) is independent of the tests that precede and follow.</p>
<h2 id="issues-not-covered-in-this-test-process">Issues Not Covered in This Test Process</h2>
<p>Problems may be found during testing that may affect accessibility, but are simply coding errors and often affect general usability for all users. An example might include a link that leads to the wrong target website. Testers may notify a developer of these issues as a comment on a report, but they do not typically result in a compliance failure as they are beyond the scope of the Revised Section 508 Standards</p>
<h2 id="the-rationale-for-each-test">The Rationale for Each Test</h2>
<p>Previous versions of the Trusted Tester process document provided a rationale for each test based on interpretation of the Section 508 standards. With the Section 508 standards refresh and adoption of the WCAG 2.0 Success Criteria, this version of the test process relies principally on the rationale provided in <a href="https://www.w3.org/TR/UNDERSTANDING-WCAG20/"><em>Understanding WCAG 2.0: A guide to understanding and implementing Web Content Accessibility Guidelines 2.0</em></a>. The test process also relies on accompanying Trusted Tester training to provide additional description and guidance for understanding the logic that drives each test. Each step included in this test process document includes only the information necessary to execute the test. However, the Applicable Standards section of each test references the applicable WCAG 2.0 or Section 508 standard along with a link to the applicable article from the <em>Understanding WCAG 2.0</em> document.</p>

<h1 id="conformance-reporting-requirements">Conformance Reporting Requirements</h1>
<p>Test outcomes or results are the primary output from conducting Section 508 conformance testing. Trusted Tester results may have multiple audiences including developers, purchasers, internal IT management personnel, and IT project management. Each audience has different uses for Trusted Tester results so a sufficient set of information must be included to support all audiences to the extent possible. Given that this Trusted Tester process provides a set of evaluations which can be used to determine WCAG 2.0 Level A and AA conformance, Trusted Tester results may also be used outside the U.S. Federal Section 508 conformance scope. However such use, while not incompatible with the Trusted Tester process, is not the primary purpose of this document.</p>
<p>One of the primary objectives of the Section 508 law is to promote improved IT accessibility based on selection of “more accessible” over “less accessible” ICT over time by Federal agencies. Consistent, well documented use of the <a href="https://www.itic.org/policy/accessibility/vpat">Accessibility Conformance Report (ACR)</a> (update to the VPAT) format from the IT Industry Council (ITI) supports evaluating overall conformance to make such selections and therefore supports the primary objective of the Section 508 law. Trusted Tester results must be provided at minimum following the Accessibility Conformance Report format from the IT Industry consortium. However, the ACR format must be supplemented with specific Trusted Tester test outcomes, which must then be aggregated to determine the “supported” and “not supported” outcomes for individual WCAG Success Criteria results.</p>
<p>Each ACR must provide:</p>
<ul>
<li><p>Clear identification of the test process used to return conformance results.</p></li>
<li><p>Clear indication of the scope of testing.</p></li>
<li><p>Clear documentation of the test environment(s).</p></li>
</ul>
<p>In general, individual Trusted Test results should provide specific Trusted Test process outcomes: <strong>PASS, FAIL, DOES NOT APPLY,</strong> or <strong>NOT TESTED.</strong> For <strong>FAIL</strong> results should also include clear information identifying the location of the failure and, when feasible, clear information illustrating the content or information that resulted in the <strong>FAIL</strong> result. When multiple instances of the same failure are found, they may be flagged as global or included individually within test results. The test results must include at minimum a reference to the cross-reference appendix in this document to explain how individual Trusted Test process tests and Baseline tests affect “supported” or “not supported” results for each of the WCAG Success Criteria.</p>
<h1 id="section-508-conformance-tests">Section 508 Conformance Tests</h1>
<p><span id="_Toc516043109" class="anchor"><span id="_Toc516043115" class="anchor"></span></span>Each of the Test Conditions included in each test section below is a statement that can be evaluated as <strong>TRUE</strong> or <strong>FALSE</strong>. The “How to Test” content included under each Test Condition provides instructions on how to evaluate whether content <strong>PASSES</strong> the condition:</p>
<ul>
<li><p>If a Test Condition is <strong>TRUE</strong> for the content in question, then the content <strong>PASSES</strong> that Test Condition.</p></li>
<li><p>If a Test Condition is <strong>FALSE</strong> for the content in question, then the content <strong>FAILS</strong> that Test Condition.</p></li>
<li><p>A Test Condition <strong>DOES NOT APPLY (DNA)</strong> only if content does not exist that meets the conditions described in the “Identify Content” instructions at the beginning of each test section.</p></li>
<li><p>In very limited instances specifically described in this test process, a Test Condition is <strong>NOT TESTED</strong> if content exists that meets the Identify Content criteria for a test, but a test cannot be performed due to limitations in available test tools.</p></li>
</ul>
<h2 id="conforming-alternate-version-and-non-interference">1. Conforming Alternate Version and Non-Interference</h2>
<p>The Conforming Alternate Version Test Conditions deviate slightly from the remaining tests in a few important ways:</p>
<ul>
<li><p>The Conforming Alternate Version Test Conditions are presented first because identifying conforming accessible alternate versions of content helps define the scope of testing. Non-conforming content that has a conforming alternate version is excluded from testing.</p></li>
<li><p>While it is important to identify conforming alternate versions to define the test scope, it will not be possible to determine if a conforming alternate version was provided until all other applicable tests are completed.</p></li>
<li><p>The results for Conforming Alternate Version Test Conditions do not follow the <strong>TRUE=PASS, FALSE=FAIL, DOES NOT APPLY (DNA)</strong> pattern that the rest of the Test Conditions follow. Providing a conforming alternate, accessible version is not a requirement. Rather, it is an acceptable method to provide the same information and/or functionality in an accessible manner when other non-conforming version(s) of that content exist. Therefore, <strong>FAIL</strong> is not an appropriate test result. When a Test Condition is <strong>TRUE</strong>, the Test Condition result is <strong>PASS</strong>. If a Test Condition is <strong>FALSE</strong>, a conforming alternate version does not exist and the Test Condition result is <strong>DOES NOT APPLY</strong>.</p></li>
</ul>
<h3 id="alternate-version-accessibility">Alternate Version Accessibility</h3>
<h4 id="identify-content">Identify Content</h4>
<p>Identify multiple versions of the same content, i.e. content that has been provided in more than one way and with the explicit or implicit intent that one or more is provided as an accessible alternative.</p>
<ol style="list-style-type: decimal">
<li><p>Alternate versions may be provided for a part of the page, entire pages, or an entire site.</p></li>
<li><p>Various methods may indicate that an alternate version is available, including:</p>
<ul>
<li><p>Instructions that describe how to enable accessibility</p></li>
<li><p>Content is identified as the accessible version</p></li>
<li><p>Multiple methods are provided to complete a task (e.g. a calendar widget and a text field are provided for a user to enter a date)</p></li>
<li><p>A link’s destination is an accessible alternate version or a version for assistive technology (e.g. screen reader version)</p></li>
<li><p>User preferences or settings to enable accessibility</p></li>
<li><p>User controls to modify colors and text appearance</p></li>
</ul></li>
<li><p>The alternate version(s) does not need to reside within the scope of conformance, or even on the same web site, as long as it is as freely available as the non-conforming version. For this test process, scope of testing is limited to alternatives that are available on a desktop computer. Alternate versions do not include mobile applications that can only be accessed on a mobile device.</p></li>
</ol>
<p>If there is only one version of content, the result for the following test ID(s) is <strong>DOES NOT APPLY</strong>: 1.A to 1.E.</p>

<h2 id="repetitive-content">9. Repetitive Content</h2>

<h2 id="version-5.0-november-2018">Version 5.0, November 2018</h2>
<p>Version 5.0 is a wholesale change to the test process and supporting content. The primary change to the document includes significant changes to the test processes to address the Section 508 Refresh and incorporation of the WCAG 2.0 Level A and Level AA Success Criteria. However, the document also includes a number of other changes to improve the organization and flow of the document, test condition logic, and readability of test process instructions. These changes include:</p>
<ul>
<li><p>General alignment of Test Condition construction with the draft W3C Accessibility Conformance Testing (ACT) Task Force Test Rules Format.</p></li>
<li><p>Removed references to Failure Conditions and reconstructed as Test Conditions with positive pass/fail construction to eliminate double negatives.</p></li>
<li><p>More straightforward Test Conditions and explanation for when tests do not apply. The revised test process includes conditions for identifying content to test. If those conditions are not met, the test does not apply.</p></li>
<li><p>Test sections re-grouped and/or renamed to use terms that are not mutually exclusive.</p></li>
<li><p>Clearer alignment of “How to Test” instructions to each Test Condition.</p></li>
<li><p>Addition of unique and descriptive test names to help identify the Test Condition without having to look up the Test Condition by Test ID.</p></li>
</ul>
<p>Finally, the test process has adopted a new testing tool: the Accessible Name and Description Inspector (ANDI). ANDI replaces all of the necessary functionalities previously provided by the Web Accessibility Favelets (WAF). ANDI also adds a significant number of new functions and features to facilitate the Trusted Tester process.</p>
